# JarPorter 优化 Backlog 规格

**日期**: 2026-07-11  
**状态**: 已确认结构（策略 C）— 只盘点、不定实施计划  
**范围**: `jar-to-harbor/` 主应用  
**非目标**: 本轮不写业务代码、不排 sprint、不定「先做哪条」的工期

---

## 1. 背景与目标

JarPorter（ShipForge）是 Tauri 2 桌面应用：JAR/dist 构建推送 Harbor、分支打包、落地页、结算、打包加速等。Rust 侧已按模块拆分（`lib.rs` 较薄），但前端根组件与若干后端文件体量偏大，测试与安全配置存在明显短板。

**目标**: 固化一份可引用的优化 backlog（`OPT-ID`），供后续按条进入 `writing-plans` / GSD phase，避免无 ID 的「顺便大重构」。

**消费方式**:

1. 从本文选 **OPT-ID**
2. 单独开实现计划，只针对该 ID（或明确声明的一小簇相关 ID）
3. 完成后在本文或关联 plan 中标记 done
4. **禁止**无 ID 的全仓大扫除

## 3. 优先级定义

| 级 | 含义 | 是否本 backlog「默认可动」 |
|----|------|---------------------------|
| **P0** | 安全/密钥与敏感信息 | **OPT-001、OPT-002 冻结暂不实施**；OPT-003 可做 |
| **P1** | 可维护性 / 架构债（体量与边界） | 可做 |
| **P2** | 测试与质量门 | 可做 |
| **P3** | 性能与体验 | 可做 |
| **P4** | 工程一致性与文档边界 | 可做 |

优先级是**风险/价值标签**，不是实施排期。排期在选中 ID 后的 plan 阶段决定。

## 4. 条目格式

每条固定字段：

- **优先级** / **区域** (`frontend` | `backend` | `security` | `tooling` | `cross`)
- **证据** / **问题** / **建议方向**
- **依赖** / **验收线索**
- **实施状态**: `frozen` | `open` | `done`（本文件初始：001/002 为 `frozen`，其余为 `open`）

| 字段 | 内容 |

| 依赖 | 无 |
| 验收线索 | 源码与默认产物中无 FTP 明文密码；上传仍可通过用户配置的凭证工作 |

#### OPT-002 Harbor 密码持久化方式

| 字段 | 内容 |
|------|------|
| 优先级 | P0 |
| 区域 | security / cross |
| 状态 | **frozen（暂不实施）** |
| 证据 | `HarborConfig.password` 与 `load_config` / `save_config` 持久化路径 |
| 问题 | 密码可能以明文 JSON 落在用户配置目录 |
| 建议方向（冻结说明） | 审计落盘形态；改为 OS 钥匙串或配置内密文 + 本机密钥；旧配置迁移或明确一次性重填；**同样禁止**把可逆密钥写进客户端源码冒充安全 |
| 依赖 | 无 |
| 验收线索 | 文档写清存储形态；磁盘上不可直接读出明文密码（在选定方案下） |
| 备注 | 桌面端无法防御本机已授权用户抠内存；目标是防误提交、日志泄露、配置文件被顺手扫一眼 |
