# 分支打包镜像分别复制设计

## 背景

当前分支打包在 npm 项目勾选“同时打包后端”并联动推送 Harbor 后，会得到前端镜像和后端镜像两条结果。页面展示时把它们合并到 `branchFullImage` 多行字符串中，并且复制按钮会一次性复制整个字符串，导致用户无法只复制前端镜像或只复制后端镜像。

## 目标

- npm 只推前端时，展示一行前端镜像，并只复制前端镜像地址。
- npm 同时推后端时，展示两行镜像：前端一行、后端一行，每行独立复制。
- Maven 项目保持单个镜像展示和复制能力。
- 复制内容必须是纯镜像地址，不包含“前端:”或“后端:”标签，也不能把两个镜像拼在一起。

## 非目标

- 不改变打包、构建镜像、推送 Harbor 的后端流程。
- 不改变镜像命名规则和标签生成规则。
- 不新增“复制全部”按钮。
- 不重构历史记录展示。

## 推荐方案

在前端把分支推送结果结构化保存，避免继续把展示标签和镜像地址混在一个字符串里。

新增一个分支镜像展示数据结构，例如：

```ts
type BranchImageResult = {
  role: "frontend" | "backend";
  label: string;
  image: string;
};
```

`App.tsx` 在自动推送成功后维护 `branchImageResults: BranchImageResult[]`：

- Maven 项目推送成功后写入 `[{ role: "backend", label: "后端镜像", image }]`。
- npm 前端推送成功后写入 `[{ role: "frontend", label: "前端镜像", image }]`。
- npm 同时打包后端且后端推送成功后追加 `{ role: "backend", label: "后端镜像", image }`。
- 开始新一轮分支打包时清空该数组。

`BranchPanel.tsx` 接收 `branchImageResults`，并按数组渲染多行：

- 每行左侧显示标签。
- 中间显示对应镜像地址。
- 右侧按钮文案使用 `复制前端` 或 `复制后端`。
- 点击按钮调用现有 `onCopyImage(image)`，只传当前行的纯镜像地址。

## 数据流

1. 用户在分支打包页选择 npm 项目，勾选“同时打包后端”和“打包后联动推送镜像到 Harbor”。
2. `handlePackageFromBranch` 先调用 `package_from_branch` 获取前端产物和可选后端产物。
3. 前端镜像推送成功后，从 `build_and_push` 返回文本中提取完整镜像地址，写入 `branchImageResults` 的前端项。
4. 如果存在后端产物，继续推送后端镜像；成功后写入后端项。
5. `BranchPanel` 根据 `branchImageResults` 展示一行或两行镜像，并分别复制。

## 错误处理

- 前端推送失败时，不展示前端镜像项，保留现有失败日志。
- 后端推送失败但前端成功时，只展示前端镜像项，日志继续提示后端失败。
- 复制仍复用现有 `handleCopyImage`，保持 toast 和 copied 状态行为。

## 测试计划

- 为镜像结果渲染增加组件级或可测试的纯函数覆盖：给定前端/后端两个镜像，复制前端按钮只传前端镜像，复制后端按钮只传后端镜像。
- 运行 TypeScript 构建检查，确认新增类型和组件 props 无错误。
- 运行现有前端测试或可用的项目验证命令；若项目没有测试脚本，则至少运行 `npm run build`。

## 自检

- 内容完整，没有未决条目。
- 范围只覆盖分支打包推送后的镜像展示和复制。
- 结构化数据是实现细节，不要求 Rust 返回值变化。
- npm 前端、npm 前端加后端、Maven 单镜像三种路径均有明确行为。
